# Architecture/Data Flow Audit - 2026-04-27

Объект: `DE_project` / AgentFlow.  
Метод: статический аудит кода и документации; сервисы и внешняя инфраструктура не поднимались.  
Фокус: границы `src/serving`, `src/processing`, `src/ingestion`, SDK и интеграций.

## 1. Карта модулей

| Зона | Ответственность по факту | Основные публичные точки |
|---|---|---|
| `src/ingestion` | Схемы входных событий, Kafka producer для synthetic/raw events, Debezium config, CDC normalization, tenant routing config | `OrderEvent`, `PaymentEvent`, `ClickstreamEvent`, `ProductEvent`, `CdcEvent`; `normalize_debezium_event`; `DebeziumConnectorConfig`; `MySqlDebeziumConnectorConfig`; `TenantRouter` |
| `src/processing` | Local pipeline, Flink jobs, shared validation/enrichment orchestration, Iceberg local sink, dead-letter replay/outbox | `local_pipeline.run`; `ValidateAndEnrich`; `SessionAggregator`; `IcebergSink`; `EventReplayer`; `OutboxProcessor` |
| `src/serving` | FastAPI app, semantic catalog/query engine, auth/rate limit, cache, observability, control-plane endpoints, DuckDB/ClickHouse backends | `/v1/*` routes; `DataCatalog`; `QueryEngine`; `ServingBackend`; `DuckDBBackend`; `ClickHouseBackend`; `QueryCache`; auth/admin APIs |
| Python SDK `sdk/agentflow` | Typed sync/async client over HTTP API, resilience, CLI helpers | `AgentFlowClient`, `AsyncAgentFlowClient`, `OrderEntity`, `MetricResult`, `QueryResult`, `HealthStatus`, CLI `agentflow` |
| TypeScript SDK `sdk-ts` | Typed ESM client over the same HTTP API, SSE support | `AgentFlowClient`, exported models, `streamEvents`, retry/circuit breaker |
| `integrations` | Framework adapters over Python SDK | LangChain tools, LlamaIndex reader/tool spec, CrewAI tools, MCP server tools |

Clean boundaries:
- `ingestion` does not import `processing` or `serving`.
- `processing` imports `ingestion` and `quality`, but not `serving`.
- SDKs do not import `src.*`; integrations depend on the Python SDK rather than duplicating HTTP logic.

Leaky boundaries:
- `serving` imports `processing` in API startup, telemetry, dead-letter replay, and NL query tracing.
- `serving.semantic_layer.query` imports `src.ingestion.tenant_router`, so serving depends on an ingestion-owned config object.
- Control-plane write/replay behavior (`EventReplayer`, `OutboxProcessor`) is split between `serving` routers and `processing` classes.

## 2. Карта потоков данных

### 2.1 Raw events path

`src/ingestion/producers/event_producer.py` generates order, payment, clickstream, and product events using the canonical Pydantic models in `src/ingestion/schemas/events.py`.

Flow:

```text
producer -> Kafka raw topics -> Flink stream_processor
  -> JSON parse
  -> optional Debezium normalization
  -> schema validation
  -> semantic validation
  -> enrichment
  -> deduplication by event_id
  -> Kafka events.validated or events.deadletter
```

Code anchor: `src/processing/flink_jobs/stream_processor.py` consumes raw and CDC topics, normalizes Debezium records, validates with `validate_event`, runs `validate_semantics`, enriches domain events, and sinks to `events.validated` / `events.deadletter`.

### 2.2 CDC path

Postgres and MySQL connector configs are generated by:
- `src/ingestion/connectors/postgres_cdc.py`
- `src/ingestion/connectors/mysql_cdc.py`

Debezium envelopes are normalized by `src/ingestion/cdc/normalizer.py` into the canonical CDC event shape:

```text
Debezium before/after/source/op
  -> normalize_debezium_event
  -> {event_id, event_type, operation, source, entity_type, entity_id, before, after, source_metadata}
  -> Flink validation path
```

Supported mapped CDC tables:
- `orders_v2`
- `users_enriched`
- `products_current`
- `sessions_aggregated`

### 2.3 Local demo path

`src/processing/local_pipeline.py` bypasses Kafka/Flink and runs the same validation/enrichment functions locally.

```text
synthetic event generation
  -> validate_event
  -> validate_semantics
  -> enrich_order / enrich_clickstream / compute_payment_risk_score
  -> DuckDB serving tables
  -> optional IcebergSink tables
  -> pipeline_events audit rows
```

DuckDB materialized serving tables:
- `orders_v2`
- `users_enriched`
- `products_current`
- `sessions_aggregated`
- `pipeline_events`

Optional Iceberg tables via `IcebergSink`:
- `orders`
- `payments`
- `clickstream`
- `inventory`
- `dead_letter`

### 2.4 Serving read path

`src/serving/api/main.py` initializes:

```text
DataCatalog
  -> DuckDBPool
  -> QueryEngine
  -> DuckDBBackend or ClickHouseBackend
  -> SearchIndex
  -> QueryCache
  -> AuthManager
  -> WebhookDispatcher / AlertDispatcher / OutboxProcessor
```

Agent-facing reads:

```text
HTTP /v1/entity, /v1/metrics, /v1/query
  -> auth/versioning/analytics/security/correlation middleware
  -> QueryEngine
  -> DataCatalog entity/metric definitions
  -> ServingBackend SQL execution
  -> PII masking and API version response transform
  -> SDK / integration caller
```

Serving backends currently implemented in code:
- `duckdb`
- `clickhouse`

The documented production path references Iceberg/Trino/Athena, but the checked serving backend factory does not expose Trino/Athena/Iceberg read backends.

### 2.5 Control-plane and reverse flows

Operational flows are served from the same FastAPI process:

```text
/v1/stream/events -> polling pipeline_events -> SSE
/v1/webhooks -> config YAML + pipeline_events polling -> signed HTTP callbacks
/v1/alerts -> metric evaluation -> webhook-style alert delivery + alert history
/v1/deadletter -> dead_letter_events -> EventReplayer -> outbox -> Kafka events.raw
/v1/slo -> pipeline_events aggregate checks
/v1/admin/* -> AuthManager + api_usage DuckDB table
```

This makes the API both read boundary and operational control plane.

### 2.6 SDK and integrations flow

```text
Python SDK / TypeScript SDK
  -> HTTP v1 API
  -> typed response models
  -> retry and circuit breaker
  -> LangChain / LlamaIndex / CrewAI / MCP adapters
```

The integrations are thin wrappers over the Python SDK:
- LangChain: `OrderLookupTool`, `MetricQueryTool`, `NLQueryTool`
- LlamaIndex: `AgentFlowReader`, `AgentFlowToolSpec`
- CrewAI: `get_agentflow_tools`
- MCP: tools `entity_lookup`, `metric_query`, `nl_query`, `health_check`, `list_entities`

## 4. Риски архитектуры

### R1. Serving boundary смешивает read API, control plane и processing write paths

Severity: medium-high.

Evidence:
- `src/serving/api/main.py` starts `OutboxProcessor`.
- `src/serving/api/routers/deadletter.py` calls `EventReplayer`.
- `src/serving/semantic_layer/query/engine.py` imports `TenantRouter` from ingestion.
- `src/serving/api/telemetry.py` and query mixins import processing tracing helpers.

Impact: API process availability and release cadence become tied to Kafka replay, DuckDB write tables, processing retry semantics, and ingestion config. This makes it harder to scale or deploy serving independently from processing/control-plane responsibilities.

Recommendation: move tenant/runtime config to a shared `src/runtime` or `src/common` module; expose replay/outbox through a processing service/interface; keep FastAPI as adapter over explicit control-plane ports.

### R2. Documented production flow is ahead of implemented serving/storage path

Severity: high.

Docs describe `Kafka -> Flink -> Iceberg -> Semantic Layer -> Agent API`, but code shows:
- `stream_processor.py` writes validated/dead-letter events back to Kafka topics.
- `IcebergSink` is used by `local_pipeline.py`; no production Flink-to-Iceberg sink is wired in this file.
- `create_backend` supports only DuckDB and ClickHouse, not Trino/Athena/Iceberg.

Impact: the real production materialization path from `events.validated` into queryable serving tables is under-specified in code. Without an explicit writer or backend, the semantic layer can be correct locally while production freshness and lineage depend on out-of-band infrastructure not enforced by repo contracts.

Recommendation: add an explicit materializer contract: validated Kafka topic -> serving store/Iceberg tables -> QueryEngine backend. Either implement the production read backend or update architecture docs to name the actual serving store.

### R3. NL query safety guard is bypassed on the primary `/v1/query` path

Severity: high.

Evidence:
- `execute_nl_query` validates translated SQL with `validate_nl_sql`.
- `paginated_query` wraps translated SQL directly in `SELECT * FROM ({sql}) ...` and its comment says prevalidated, but there is no call to `validate_nl_sql` in that method.
- The API uses `paginated_query` when the engine has it, so normal `POST /v1/query` traffic takes the paginated path.

Impact: a future LLM translation regression or prompt-injection edge case can reach backend execution without the same AST allowlist applied to non-paginated execution.

Recommendation: centralize SQL preparation in one method that always performs translation, AST validation, tenant scoping, and pagination wrapping in that order. Add a regression test for unsafe SQL through `paginated_query`.

### R4. Public contract surface is fragmented across too many sources

Severity: medium.

Current contract sources include Pydantic event models, entity YAML files, schema version YAML files, API version YAML, FastAPI router models, partial `docs/openapi.json`, narrative `docs/api-reference.md`, Python SDK models, and TypeScript SDK interfaces.

Specific drift points:
- `docs/openapi.json` covers core paths only, while the code and `docs/api-reference.md` expose a much larger v1 surface.
- Python sync and TypeScript clients support contract version filtering; Python async client does not expose `contract_version`.
- Operational/admin/governance routes are public HTTP routes but mostly not first-class SDK methods.

Impact: API consumers can receive behavior that is documented narratively but not enforced by the machine-readable spec or SDK parity tests. This is manageable now, but becomes risky as control-plane routes mature.

Recommendation: split specs intentionally: `core.openapi.json` and `control-plane.openapi.json`, or publish one full generated OpenAPI artifact. Generate SDK model/type checks from the chosen spec and add parity tests for sync Python, async Python, and TypeScript.

### R5. Operational event tables are implicit and divergent

Severity: medium.

Evidence:
- `pipeline_events` is created in multiple places with different columns.
- Stream, SLO, lineage, webhook dispatch, and metrics infer columns dynamically.
- Local invalid events write to `pipeline_events` and optional Iceberg `dead_letter`, while `/v1/deadletter` reads `dead_letter_events`.
- Lineage requires `entity_id` in `pipeline_events`; the local and demo table definitions may not have it.

Impact: operational APIs can be green at the route level but return sparse, empty, or inconsistent results depending on which writer initialized the database. This weakens lineage, dead-letter replay, webhook filtering, and SLO evidence.

Recommendation: introduce one canonical operational envelope/table migration for `pipeline_events`, `dead_letter_events`, and `outbox`. Make local pipeline, Flink/materializer, and API control-plane use the same schema contract.

## 5. Summary

The highest-value architectural fix is to make the actual production materialization path explicit and contract-tested. The second highest is to close the SQL guard gap in paginated NL queries. After that, consolidate operational table schemas and clarify the boundary between serving and processing/control-plane work.

Overall shape is coherent for a reference platform: ingestion owns event contracts, processing owns validation/enrichment, serving owns semantic HTTP reads, and SDK/integrations stay thin. The main risks come from production-path ambiguity and contract drift as the API surface expanded beyond the original core agent reads.

Дата: 2026-04-27  
Scope: `D:\DE_project`, HEAD `4a13d36`  
Фокус: FastAPI auth, permissions, tenant isolation, API control-plane routes.  
Метод: статический code trace по `src/serving/api`, `src/serving/semantic_layer`, `src/processing`; live HTTP-запросы не выполнялись.

## Attack Surface

- Auth middleware: `src/serving/api/auth/middleware.py`
- API key/key rotation: `src/serving/api/auth/manager.py`, `src/serving/api/auth/key_rotation.py`
- Core API: `/v1/query`, `/v1/entity`, `/v1/metrics`, `/v1/batch`, `/v1/search`, `/v1/stream/events`
- Control plane: `/v1/webhooks`, `/v1/alerts`, `/v1/deadletter`, `/v1/lineage`, `/v1/slo`
- Admin: `/v1/admin/*`, `/admin/*`

## Findings Summary

| # | Severity | Confidence | Area | Finding |
|---|---|---:|---|---|
| 1 | CRITICAL | 9/10 | Tenant isolation | Webhook dispatcher can deliver every tenant's pipeline events to every active tenant webhook |
| 2 | CRITICAL | 9/10 | Tenant isolation | Dead-letter API is not tenant-scoped and exposes payloads plus replay/dismiss actions |
| 3 | HIGH | 9/10 | Tenant isolation | Stream/lineage/SLO endpoints read global pipeline events without tenant context |

## Finding 1: Tenant Webhooks Receive Global Pipeline Events

**Severity:** CRITICAL  
**Confidence:** 9/10  
**Status:** VERIFIED by code trace  
**Files/lines:** `src/serving/api/webhook_dispatcher.py:199`, `src/serving/api/webhook_dispatcher.py:201`, `src/serving/api/webhook_dispatcher.py:209`, `src/serving/api/webhook_dispatcher.py:302`, `src/serving/api/routers/webhooks.py:29`

**Evidence:** webhook registrations are tenant-tagged at create time, but dispatcher loads all active webhooks and fetches one global `pipeline_events` stream:

- `src/serving/api/routers/webhooks.py:29-35` stores `tenant=_tenant(request)`.
- `src/serving/api/webhook_dispatcher.py:199-211` loads all active webhooks, fetches all pipeline events once, then iterates every event over every webhook.
- `src/serving/api/webhook_dispatcher.py:289-303` reads `SELECT * FROM pipeline_events` with no tenant schema or tenant filter.

**Exploit scenario:** a malicious tenant creates `/v1/webhooks` with no filters. When another tenant produces order/user/session events, `dispatch_new_events()` posts those event bodies to the malicious tenant's URL because `_matches_filters()` only checks event fields, not `webhook.tenant`.

**Impact:** cross-tenant event exfiltration, including any fields present in `pipeline_events`. The webhook signature does not mitigate this; it authenticates the sender to the attacker-controlled receiver.

**Remediation:**
- Add a tenant identifier to pipeline events, or store pipeline events in tenant schemas.
- In `dispatch_new_events()`, deliver only events whose tenant matches `webhook.tenant`.
- If event tenant cannot be determined, fail closed and skip delivery.
- Add regression tests with two tenants and two webhooks proving tenant A never receives tenant B events.

## Finding 2: Dead-Letter API Is Global And Exposes Payloads

**Severity:** CRITICAL  
**Confidence:** 9/10  
**Status:** VERIFIED by code trace  
**Files/lines:** `src/serving/api/routers/deadletter.py:94`, `src/serving/api/routers/deadletter.py:137`, `src/serving/api/routers/deadletter.py:229`, `src/serving/api/routers/deadletter.py:244`, `src/serving/api/routers/deadletter.py:264`, `src/serving/api/routers/deadletter.py:291`

**Evidence:** every dead-letter query reads `dead_letter_events` directly and never filters by tenant:

- stats/list endpoints query all failed events at `deadletter.py:94-204`.
- detail endpoint returns `payload` for any `event_id` at `deadletter.py:229-255`.
- replay/dismiss mutate by `event_id` at `deadletter.py:264-295`.
- `_require_deadletter_write_access()` at `deadletter.py:83-91` checks only whether `allowed_entity_types` is `None`; it does not check tenant ownership.

**Exploit scenario:** tenant A calls `/v1/deadletter` and receives event IDs, failure details, and timestamps for tenant B. Then tenant A calls `/v1/deadletter/{event_id}` and receives the failed event payload. A broad tenant key can also replay or dismiss another tenant's event if it knows the ID.

**Impact:** cross-tenant payload disclosure and potential cross-tenant operational tampering.

**Remediation:**
- Persist `tenant_id` on `dead_letter_events`.
- Apply `WHERE tenant_id = request.state.tenant_id` to stats, list, detail, replay, and dismiss.
- Replace the current `allowed_entity_types is None` write gate with explicit scopes, for example `deadletter:read` and `deadletter:write`.
- Add tests for list/detail/replay/dismiss using two tenants and shared-looking event IDs.

## Finding 3: Stream, Lineage, And SLO Endpoints Ignore Tenant Context

**Severity:** HIGH  
**Confidence:** 9/10  
**Status:** VERIFIED by code trace  
**Files/lines:** `src/serving/api/routers/stream.py:22`, `src/serving/api/routers/stream.py:35`, `src/serving/api/routers/lineage.py:68`, `src/serving/api/routers/lineage.py:87`, `src/serving/api/routers/slo.py:90`, `src/serving/api/routers/slo.py:135`

**Evidence:**
- `/v1/stream/events` reads `pipeline_events` directly from `request.app.state.query_engine._conn`, with no `request.state.tenant_id` use.
- `/v1/lineage/{entity_type}/{entity_id}` fetches matching events by `entity_id` only from global `pipeline_events`.
- `/v1/slo` computes latency/freshness/error-rate over global `pipeline_events`.

**Exploit scenario:** tenant A subscribes to `/v1/stream/events` or queries `/v1/lineage/order/{known_or_guessed_id}` and sees event IDs, entity IDs, topics, timestamps, and latency data from tenant B. `/v1/slo` also discloses aggregate health across all tenants rather than the caller's tenant.

**Impact:** cross-tenant operational metadata disclosure. Depending on event ID/entity ID semantics, this can leak customer activity, volume, failure patterns, and source-system details.

**Remediation:**
- Route these endpoints through `QueryEngine` tenant-aware table qualification instead of direct `query_engine._conn`.
- Add tenant filters to `pipeline_events` queries or use tenant schemas consistently.
- Apply `allowed_entity_types` to `stream` and `lineage` entity filters.
- Add tenant-isolation tests for stream, lineage, and SLO.

## Finding 4: Entity Permissions Are Enforced Only On Direct Entity Reads

**Severity:** HIGH  
**Confidence:** 9/10  
**Status:** VERIFIED by code trace  
**Files/lines:** `src/serving/api/auth/middleware.py:102`, `src/serving/api/routers/agent_query.py:139`, `src/serving/api/routers/agent_query.py:399`, `src/serving/api/routers/batch.py:147`, `src/serving/api/routers/search.py:51`, `src/serving/semantic_layer/query/nl_queries.py:173`

**Evidence:** `allowed_entity_types` is enforced only when `_entity_type_from_path()` matches `/v1/entity/{entity_type}/...`.

- `auth/middleware.py:102-103` checks entity permissions only for direct entity path matches.
- `/v1/query` passes tenant ID to the engine but does not restrict tables by `tenant_key.allowed_entity_types`.
- `nl_queries.py:173-176` allows every catalog table plus `pipeline_events`.
- `/v1/metrics/{metric_name}` does not map metrics to allowed entity/table permissions.
- `/v1/batch` enforces entity restrictions for `type="entity"` but not for `type="query"` or `type="metric"`.
- `/v1/search` accepts caller-supplied `entity_types` but does not intersect them with key permissions.

**Exploit scenario:** a key configured for `allowed_entity_types: ["order"]` is blocked from `GET /v1/entity/user/...`, but can call `POST /v1/query` with "top products", "active sessions", or a user question that the NL layer translates to allowed catalog tables. The same key can use `/v1/search` to discover product/session/user snippets and endpoints.

**Impact:** horizontal permission bypass inside a tenant; entity-scoped keys are not actually scoped across the API surface.

**Remediation:**
- Introduce a central authorization helper that maps route/action to required entity types or scopes.
- In NL-to-SQL, restrict the LLM schema prompt and `allowed_tables` to the caller's allowed entity tables.
- Map metrics to source entity/table permissions before executing.
- Intersect search results and requested `entity_types` with `tenant_key.allowed_entity_types`.
- Add tests for denied query/search/metric/batch-query access using a restricted key.

## Finding 5: API Key Misconfiguration Fails Open

**Severity:** HIGH  
**Confidence:** 8/10  
**Status:** VERIFIED by code trace  
**Files/lines:** `src/serving/api/auth/middleware.py:34`, `src/serving/api/auth/manager.py:330`, `src/serving/api/auth/manager.py:338`, `src/serving/api/main.py:205`

**Evidence:** non-admin auth is skipped when no keys are loaded:

- `auth/middleware.py:34-35` lets every non-admin route through when `not manager.has_configured_keys()`.
- `manager.py:330-338` returns an empty config if the configured key file is missing.
- `main.py:205-209` labels this mode as `open (set config/api_keys.yaml to enable)`.

**Exploit scenario:** a production deployment has a missing Secret mount, empty `api_keys.yaml`, or unset `AGENTFLOW_API_KEYS_FILE`. The API starts and serves `/v1/query`, `/v1/entity`, `/v1/deadletter`, `/v1/webhooks`, and other non-admin routes without API-key authentication.

**Impact:** full unauthenticated access to non-admin API and tenant control-plane routes under a common deployment misconfiguration.

**Remediation:**
- Fail closed by default when not in explicit demo/local mode.
- Require an explicit flag such as `AGENTFLOW_ALLOW_OPEN_AUTH=true` for unauthenticated local development.
- If `AGENTFLOW_API_KEYS_FILE` is set but missing or has zero keys, fail startup.
- Add a deployment smoke test that starts with a missing key file and expects startup failure or `503`, not open API.

## Finding 6: Tenant API Keys Can Mutate Webhooks And Alerts Without Write Scopes

**Severity:** HIGH  
**Confidence:** 8/10  
**Status:** VERIFIED by code trace  
**Files/lines:** `src/serving/api/routers/webhooks.py:29`, `src/serving/api/routers/webhooks.py:46`, `src/serving/api/routers/alerts.py:64`, `src/serving/api/routers/alerts.py:88`, `src/serving/api/routers/alerts.py:106`, `src/serving/api/auth/manager.py:70`

**Evidence:** API keys have `tenant`, `rate_limit_rpm`, and `allowed_entity_types`, but no action scopes or roles. Webhook and alert mutating routes rely only on the global auth middleware:

- `/v1/webhooks` create/delete/test routes do not check write permission.
- `/v1/alerts` create/update/delete/test routes do not check write permission.
- `KeyCreateRequest` has no `scopes`, `role`, or `permissions` field.

**Exploit scenario:** a low-privilege tenant key intended for read-only agent usage can create an outbound webhook or alert pointing to an attacker-controlled URL, or delete/modify existing tenant alerts/webhooks.

**Impact:** privilege escalation within a tenant and a practical exfiltration path when combined with tenant event delivery issues.

**Remediation:**
- Add explicit scopes, for example `entity:read`, `query:read`, `webhook:write`, `alert:write`, `deadletter:write`.
- Enforce scopes via FastAPI dependencies per router/action.
- Treat `allowed_entity_types` as data-scope only, not as write authorization.
- Add tests proving restricted keys cannot create/update/delete webhooks or alerts.

## Finding 7: Admin Surfaces Expose Reusable Secrets

**Severity:** MEDIUM

## 1. API silently becomes unauthenticated

**Evidence:** `main.py` passes `api_keys_path=os.getenv("AGENTFLOW_API_KEYS_FILE")`; if unset, `AuthManager` does not load the checked-in `config/api_keys.yaml`. The auth middleware then bypasses auth for every non-admin/non-exempt route when `not manager.has_configured_keys()`.

**Exploit scenario:** deploy `docker-compose.prod.yml` or run the API without `AGENTFLOW_API_KEYS_FILE`; attacker calls `/v1/query`, `/v1/entity/...`, `/v1/webhooks`, `/v1/alerts` with no `X-API-Key`.

**Impact:** unauthenticated data access and webhook/alert mutation.

**Remediation:** fail closed outside explicit local/dev mode. Either default to `config/api_keys.yaml` or require `AGENTFLOW_API_KEYS_FILE`; only allow open mode with an explicit `AGENTFLOW_AUTH_DISABLED=true` and never in prod.

## 2. Admin/failed-auth rate limiting is bypassable

**Evidence:** admin paths return before auth middleware records client IP or failed attempts. Separately, `_client_ip()` trusts `X-Forwarded-For` directly.

**Exploit scenario:** attacker brute-forces `/v1/admin/*` with unlimited `X-Admin-Key` guesses. For API-key guessing, attacker rotates `X-Forwarded-For` values to get a fresh failed-auth window.

**Impact:** practical brute-force surface for admin/API keys, especially combined with low-entropy keys.

**Remediation:** apply failed-auth throttling to admin dependency too; derive client IP only from trusted proxy middleware/config; ignore user-supplied `X-Forwarded-For` unless the immediate peer is trusted.

## 3. Generated API keys have low entropy

**Evidence:** `generate_key()` uses `secrets.token_hex(4)`, which is only 8 hex chars / 32 bits. `min_key_length` only validates string length, so predictable tenant/name slugs hide the weak random suffix.

**Exploit scenario:** generated key format is `af-prod-{tenant}-{name}-{8hex}`. If tenant/name are known, attacker has a 2^32 search space.

**Impact:** API keys are feasible to brute-force, especially with distributed requests or rate-limit bypasses.

**Remediation:** use at least 128 bits of random material, e.g. `secrets.token_urlsafe(32)` or `secrets.token_hex(32)`. Validate entropy/format, not only length. Rotate keys generated by the current code.

## 4. Paginated NL query skips SQL guard

**Evidence:** `/v1/query` chooses `engine.paginated_query` when present. That method imports `validate_nl_sql` but does not call it before wrapping/executing SQL. Local verification showed `SELECT * FROM api_usage` was passed into backend execution as `SELECT * FROM (SELECT * FROM api_usage) ...`.

**Exploit scenario:** with LLM SQL mode enabled, user prompt injection causes the model to emit a `SELECT` against a non-catalog table or DuckDB file-reading function. The paginated path executes it without the guard used by `execute_nl_query`.

**Impact:** unauthorized table/file reads through the API query endpoint.

**Remediation:** call `validate_nl_sql(translated_sql, allowed_tables)` in `paginated_query` before `_scope_sql()` and before pagination wrapping. Extend the guard to reject unsafe functions, not just unsafe statement types/tables.

## 5. Tracked plaintext deployment/runtime secrets

**Evidence:** tracked files contain active/default secrets: `k8s/staging/values-staging.yaml:46` admin key, `k8s/staging/values-staging.yaml:50`/`:59`/`:66` API keys, `config/webhooks.yaml:4` webhook signing secret, `docker-compose.prod.yml:162`/`:200` ClickHouse password, `docker-compose.prod.yml:253` Grafana admin password, `helm/kafka-connect/values.yaml:54`/`:58` connector DB passwords.

**Exploit scenario:** anyone with repository access uses these credentials against staging/prod-like deployments or forges webhook signatures.

**Impact:** service takeover, data access, forged webhook deliveries.

**Remediation:** revoke/rotate all listed secrets; scrub git history if the repo was shared; move real values to external secret stores or untracked runtime files; keep only `.example` placeholders in git. Add `.gitleaks.toml` or `.secretlintrc`; neither is present.

## 6. Admin key is reflected into admin HTML/DOM

**Evidence:** `_build_context()` copies `X-Admin-Key` into template context, and `admin.html` renders it as `data-admin-key`. Client JS reuses it for refresh calls.

**Exploit scenario:** any same-origin XSS, browser extension, support bundle, cached HTML, or proxy log containing the admin page can read the admin key from DOM.

**Impact:** full admin API access.

**Remediation:** do not render the admin secret into HTML. Replace header reuse with a short-lived HttpOnly/SameSite/Secure session cookie, or make partial refresh server-side without exposing the credential to JS.

## 7. Webhook/alert signing secrets are returned repeatedly

**Evidence:** webhook and alert models include `secret`, and list endpoints serialize full models with `model_dump(mode="json")`.

**Exploit scenario:** any tenant API key that can list its webhooks/alerts can retrieve all signing secrets after creation, not only at one-time setup.

**Impact:** higher blast radius for read/list access; leaked API responses allow forged delivery signatures.

**Remediation:** return secret only once at creation if required. Use response DTOs that exclude `secret` for list/read/update. Consider encrypted-at-rest storage for outbound signing secrets.

## 8. `X-Admin-Key` is not redacted from auth-failure logs

**Evidence:** redaction policy includes `Authorization` and `X-API-Key`, but admin auth uses `X-Admin-Key`. Failed non-admin auth logs pass request headers through this policy.

**Exploit scenario:** operator or client accidentally sends `X-Admin-Key` to a non-admin endpoint with an invalid/missing `X-API-Key`; the failed-auth log records the admin key.

**Impact:** admin secret exposure through application logs.

**Remediation:** add `X-Admin-Key` to default and configured sensitive headers. Prefer pattern-based redaction for header names containing `key`, `token`, `secret`, `password`, or `authorization`.

## 9. PII masking skipped for multi-entity NL SQL

**Evidence:** `mask_query_results()` returns rows unchanged when SQL maps to zero or more than one entity type. `/v1/query` and batch query responses use that function for NL SQL results. Local verification with `users_enriched JOIN orders_v2` returned `email`/`full_name` unchanged and `changed=False`.

**Exploit scenario:** authenticated user asks a join query returning user PII plus order fields. Because two entity types are detected, masking is skipped entirely.

**Impact:** PII leakage in NL query responses.

**Remediation:** apply field-level masking across every returned column using all matched entity rules. If entity mapping is ambiguous, fail closed: mask configured sensitive field names globally or reject the query.

## 10. Redis-backed rate limiter fails open

**Evidence:** Redis pipeline errors are logged and `RateLimiter.check()` returns `(True, limit, reset_at)`.

**Exploit scenario:** Redis outage/misconfiguration disables per-key rate limiting while API continues serving requests.

**Impact:** uncontrolled request volume and cost/availability risk during dependency failure.

**Remediation:** choose an explicit fail policy. For auth and expensive query endpoints, fail closed or fall back to the in-process limiter with conservative per-key/IP limits. Expose health/alerting when Redis limiter is unavailable.

## 11. Plaintext API keys can become Redis key names

**Evidence:** `_rate_limit_key()` returns `tenant_key.key` when plaintext key material exists; `RateLimiter` uses that value as the Redis sorted-set key.

**Exploit scenario:** deployments using plaintext `key:` entries, including the tracked staging values, write the actual API key into Redis keyspace/monitoring/backups.

**Impact:** API key disclosure through Redis introspection, metrics, backups, or logs.

**Remediation:** use `key_id` or `HMAC(server_secret, key)` as the rate-limit identifier. Never put raw API keys or bcrypt hashes directly into cache key names.

# Python SDK Audit - SDK/server mismatch findings

Дата: 2026-04-27  
Repo: `D:\DE_project`  
HEAD: `4a13d36`  
Scope: `sdk/agentflow`, публичный Python API, retries/timeouts, typed contracts, ошибки и совместимость с server API.

## Findings

### F1 - High - `contract_version` в Python SDK не совместим с server API versioning

Server versioning выбирается через header `X-AgentFlow-Version` или tenant pin: `src/serving/api/versioning.py:236`, `src/serving/api/versioning.py:249`, `src/serving/api/versioning.py:255`. Ответы также возвращают `X-AgentFlow-Version`, `X-AgentFlow-Latest-Version`, deprecated/warning headers: `src/serving/api/versioning.py:279`.

Python SDK вместо этого принимает `contract_version="entity:vN"` и локально фильтрует entity payload через `/v1/contracts/{entity}/{version}`: `sdk/agentflow/client.py:51`, `sdk/agentflow/client.py:171`, `sdk/agentflow/client.py:208`. HTTP client отправляет только `X-API-Key`: `sdk/agentflow/client.py:53`.

Impact: пользователь думает, что pinned SDK работает с нужной server API версией, но сервер продолжает отдавать latest или tenant-pinned date version. Это особенно опасно для `meta` transform и deprecation headers.

Fix: добавить явный параметр `api_version` или `agentflow_version`, отправлять `X-AgentFlow-Version`, читать response version headers, а `contract_version` оставить только для schema-contract filtering.

### F2 - High - Async SDK не поддерживает contract pinning вообще

Sync client имеет `contract_version` и `_apply_contract_version`: `sdk/agentflow/client.py:46`, `sdk/agentflow/client.py:171`. Async constructor принимает только `base_url`, `api_key`, `timeout`: `sdk/agentflow/async_client.py:46`, а `_get_entity` валидирует напрямую `envelope.data`: `sdk/agentflow/async_client.py:146`.

Impact: два публичных клиента дают разные typed-contract guarantees. Интеграция, которая проходит на sync SDK с `contract_version="order:v1"`, может сломаться или принять лишние поля на async SDK.

Fix: выровнять async API с sync: тот же `contract_version`, cache contracts, async `_get_contract`, единые contract tests для sync/async.

### F3 - High - Historical API (`as_of`) есть на сервере, но отсутствует в Python SDK

Server поддерживает `as_of` для entity и metric: `src/serving/api/routers/agent_query.py:247`, `src/serving/api/routers/agent_query.py:405`. Документация относит это к core API: `docs/api-reference.md:59`, `docs/api-reference.md:60`.

Python SDK методы не принимают `as_of`: `get_order/get_user/get_product/get_session` вызывают `_get_entity` без query params, а `_get_entity` строит только `/v1/entity/{type}/{id}`: `sdk/agentflow/client.py:143`, `sdk/agentflow/client.py:212`. `get_metric` передает только `window`: `sdk/agentflow/client.py:224`.

Impact: SDK не может использовать time-travel/freshness workflows, хотя сервер и docs объявляют их частью API.

Fix: добавить `as_of: datetime | str | None` в generic/entity/metric methods и нормализовать ISO UTC в query params.

### F4 - High - Серверные `meta` поля silently dropped by SDK models

Entity responses возвращают `meta.as_of`, `meta.is_historical`, `meta.freshness_seconds`: `src/serving/api/routers/agent_query.py:368`. Metric responses возвращают `meta` с historical context: `src/serving/api/routers/agent_query.py:495`.

Python models игнорируют лишние поля через `AgentFlowModel.model_config = ConfigDict(extra="ignore")`: `sdk/agentflow/models.py:7`. `EntityEnvelope` и `MetricResult` не имеют `meta`: `sdk/agentflow/models.py:11`, `sdk/agentflow/models.py:70`.

Impact: даже если добавить `as_of`, SDK потеряет серверный признак historical response. Пользователь не сможет отличить live value от historical value через typed model.

Fix: добавить `meta: dict[str, Any] = Field(default_factory=dict)` в entity envelope и metric/query response contracts, либо typed `EntityMeta`/`MetricMeta`.

### F5 - High - Catalog SDK model не соответствует server catalog

Server `/v1/catalog` возвращает `entities`, `metrics`, `streaming_sources`, `audit_sources`: `src/serving/api/main.py:314`. Entity/metric catalog также содержит `contract_version`: `src/serving/semantic_layer/catalog.py:157`, `src/serving/semantic_layer/catalog.py:168`.

Python SDK `CatalogResponse` моделирует только `entities` и `metrics`; `CatalogEntity`/`CatalogMetric` не включают `contract_version`: `sdk/agentflow/models.py:111`, `sdk/agentflow/models.py:123`.

Impact: SDK скрывает discovery/governance surface, который сервер рекламирует для agents. Через typed SDK нельзя обнаружить SSE stream, lineage/audit route и актуальные schema contract versions.

Fix: расширить `CatalogResponse` до полного server payload: `streaming_sources`, `audit_sources`, `contract_version` в entity/metric.

### F6 - Medium - Python SDK public API покрывает только часть public v1 server API

Maintained API reference перечисляет core/discovery routes: `/v1/search`, `/v1/contracts`, `/v1/contracts/{entity}`, `/v1/contracts/{entity}/{version}`, `/v1/contracts/{entity}/diff/...`, `/v1/contracts/{entity}/validate`, `/v1/lineage/...`, `/v1/changelog`: `docs/api-reference.md:69`.

Python SDK public methods ограничены entity/metric/query/health/catalog/batch helpers: `sdk/agentflow/client.py:212`, `sdk/agentflow/client.py:292`, `sdk/agentflow/client.py:296`. `query/explain` есть на server: `src/serving/api/routers/agent_query.py:126`, но SDK имеет только executing `query`: `sdk/agentflow/client.py:253`.

Impact: Python SDK не является full client для documented v1 API. Часть возможностей доступна только через private `_request` или CLI, что ломает typed contracts и error mapping.

Fix: добавить typed methods минимум для `explain_query`, `search`, contract list/get/diff/validate, `lineage`, `changelog`; явно задокументировать, какие admin/ops routes намеренно вне SDK.

### F7 - Medium - Dynamic entity registry на сервере конфликтует с hardcoded SDK entity API

Server грузит entity types из `contracts/entities/*.yaml`, и docs говорят, что новый тип доступен через `/v1/entity/{type}/{id}` без Python-кода: `docs/contracts/how-to-add-entity.md:3`, `src/serving/semantic_layer/entity_type_registry.py:58`.

Python SDK экспортирует только четыре hardcoded typed methods: `get_order`, `get_user`, `get_product`, `get_session`: `sdk/agentflow/client.py:212`. Generic `_get_entity` приватный: `sdk/agentflow/client.py:143`.

Impact: кастомная entity, поддержанная сервером и catalog, недоступна как публичный SDK call без обращения к private method или raw `_request`.

Fix: добавить public `get_entity(entity_type, entity_id, *, as_of=None) -> EntityEnvelope` и оставить typed convenience methods как thin wrappers.

### F8 - Medium - Retry helper умеет idempotent POST, но Python client не может его включить

`is_retryable_method` разрешает retry для `POST` только при наличии `Idempotency-Key`: `sdk/agentflow/retry.py:30`. Но `_request` не принимает custom headers и вызывает `is_retryable_method(method)` без headers: `sdk/agentflow/client.py:74`, `sdk/agentflow/client.py:83`.

Impact: SDK не может безопасно retry `POST /v1/query` или `POST /v1/batch`, хотя retry module уже содержит такой контракт. В результате временный 503 на query/batch сразу отдается пользователю.

Fix: добавить headers/idempotency_key в `_request` и public POST methods, либо убрать unreachable branch из retry contract и явно задокументировать no POST retries.

### F9 - Medium - Ошибки 403 и circuit-open не представлены как стабильные SDK errors

Auth middleware возвращает 403 при entity permission denial: `src/serving/api/auth/middleware.py:102`. Python SDK мапит только 401, 429, 404; остальные 4xx становятся generic `AgentFlowError`: `sdk/agentflow/client.py:121`, `sdk/agentflow/client.py:137`.

Circuit breaker может бросить `CircuitOpenError`, но этот класс наследуется от `RuntimeError`, не от `AgentFlowError`: `sdk/agentflow/circuit_breaker.py:13`; root package `__all__` его не экспортирует: `sdk/agentflow/__init__.py:6`.

Impact: callers не могут надежно различать auth failure, permission denied, rate limit, not found и local circuit-open через единое SDK error taxonomy.

Fix: добавить `PermissionDeniedError` для 403, экспортировать resilience errors из root или сделать `CircuitOpenError(AgentFlowError)`.

### F10 - Medium - Package claims typed, но wheel/sdist не содержат `py.typed`

`sdk/pyproject.toml` объявляет classifier `Typing :: Typed`: `sdk/pyproject.toml:16`. В `sdk/agentflow` нет `py.typed`, и wheel `sdk/dist/agentflow_client-1.1.0-py3-none-any.whl` содержит Python modules/templates, но не marker file.

Impact: type checkers не обязаны считать package PEP 561 typed package, несмотря на публичный classifier. Это делает typed SDK contracts менее полезными для downstream users.

Fix: добавить `sdk/agentflow/py.typed`, убедиться, что hatch включает marker в wheel/sdist, и добавить package artifact test.

## Cross-cutting recommendation

Сейчас Python SDK лучше считать partial convenience client, а не authoritative typed client for v1. Для release-quality SDK/server compatibility нужен один источник truth: OpenAPI или generated contract snapshot, из которого проверяются Python models, TS models, docs и live FastAPI routes.

1. Ввести `X-AgentFlow-Version` support и `as_of` в Python SDK.
2. Выровнять sync/async contract pinning.
3. Расширить `CatalogResponse`/`EntityEnvelope`/`MetricResult` до полного server payload.

## What Should Become Release Blockers

Block every production release on:
1. No plaintext secrets in repo release artifacts, including sdists.
2. `npm audit` must run from a committed lockfile and pass the configured severity threshold.
3. Python dependency audit must run from locked/resolved release inputs and include root, SDK, integrations, and published extras.
4. No High/Critical vulnerabilities in runtime, published extras, or release tooling unless a documented exception includes exploitability analysis and expiry.
5. Bandit diff must show no new medium/high findings relative to baseline.
6. Third-party GitHub Actions in release/security workflows must not use floating branch refs such as `master`.
7. Publish workflows must use reproducible installs: `npm ci` for npm and locked/constraints-driven Python installs for build and checks.
8. Built artifacts must pass `twine check`/`npm pack --dry-run` and an artifact content allowlist/secret scan.
