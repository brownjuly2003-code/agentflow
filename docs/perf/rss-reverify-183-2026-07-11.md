# API RSS live re-verification — fix #184 holds under soak profile (2026-07-11)

> Stand: `deproject-mac` (Colima vz 6 GiB / 4 CPU, macOS 13.7.8 Intel), code
> `main @ 9f4cfb5` (+ benchmark harness, no `src/` changes).
> This is the live re-verification promised in [STATUS.md](../STATUS.md) after
> issue #183 was fixed in #184 and proven flat at unit scale.

## Setup — deliberately harsher than the S11 soak that found the leak

| Dimension | S11 (leak found) | This run |
|---|---|---|
| Journal size at start | grew 0 → 683 k during the run | **1.371 M rows from the first second** |
| Read traffic | 12 rpm (`GET /v1/metrics/revenue`, demo key) | same |
| Webhook dispatcher | scanning every 2 s | same (incremental cursor path from #184) |
| Fresh applies | ~47 eps × 4 h via bridge | 50 eps × 60 min via bridge (journal → 1.732 M) |
| Sampler | 60 s (RSS/FD/lag/journal) | same |

The old code re-materialized the whole journal every 2 s (0.71 MB/1k rows) —
on this run's starting journal that is ~1 GB of transient allocations per
scan, so a regression would have been unmistakable within minutes.

## Result — flat

| Metric | Value |
|---|---|
| Window | **97 min**, 98 one-minute samples |
| API RSS first → last | 77.3 MB → 100.9 MB |
| OLS slope, full window | **+7.5 MB/h** |
| OLS slope, growth phase (69 samples, journal 1.374 M → 1.732 M) | **+3.2 MB/h** |
| Quartile means | 92.7 · 101.3 · 104.3 · **100.1** MB — plateaued, Q4 below Q3 |
| Max transient (GC sawtooth) | 185 MB, immediately reclaimed |
| API FDs | pinned 149–152 |
| Reads | 1 149 / 1 149 × HTTP 200, 0 errors |
| Bridge RSS / FDs | sawtooth 35–108 MB / pinned 94–95 (unchanged healthy signature) |

Reference points: the pre-fix leak measured **~+370 MB/h monotonic**
(175 MB → 1.67 GB over 4 h) on a *smaller* journal; the S11-validated alert
threshold is monotonic growth > 200 MB/h. This run sits 25–60× below the
threshold and is not monotonic. **Issue #183 is closed live, not only at unit
scale.**

## Honest deviations from the S11 profile

1. **The Flink hop was bypassed for the growth phase.** The stand VM could not
   keep the Flink TaskManager alive tonight: three TM deaths by JVM **native
   crash** (outside-VM code, RocksDB during asynchronous checkpoints to MinIO;
   47 failed / 16 completed checkpoints), a crash loop unrelated to the
   product code — the same job ran 4 h on 2026-07-10. Fresh events were
   produced directly to `events.validated` in the bridge's canonical shape
   (`tests/integration/test_serving_bridge.py::_flink_shaped_order`), so the
   measured surface — bridge apply → ClickHouse journal → dispatcher cursor
   scan → API reads — is identical; only validation upstream of the topic was
   skipped. The full-path endurance claim remains
   [S11](soak-s11-2026-07-10.md); this run re-verifies the API-side fix.
2. The first ~29 min ran against the **static** 1.371 M journal (dispatcher
   scanning, reads flowing, no applies) while the Flink crash loop was being
   diagnosed — that phase alone is the strongest repro condition for the old
   bug and is included in the full-window slope.
3. Journal row counts in the sampler CSV are raw `count()` over
   `pipeline_events` (2 rows per applied event pre-merge, a ReplacingMergeTree
   artifact also present in the S10/S11 numbers; delta 360 k = 180 k events).
4. Co-tenant containers (DataLens family) auto-restarted inside the VM during
   part of the window — ambient churn characteristic of this stand. The API
   process runs on the host, outside the VM, so its RSS is unaffected; the
   read path stayed 100 % HTTP 200 throughout.
