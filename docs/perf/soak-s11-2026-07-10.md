# S11 endurance soak — 4 h sustained real path (2026-07-10)

> Stand: `deproject-mac` (Colima vz 6 GiB / 4 CPU, macOS 13.7.8 Intel), code `main @ 8b7181e`.  
> Path: produce(`orders.raw`) → Flink 2.3.0 → `events.validated` → serving bridge → ClickHouse,
> plus steady API read traffic (`GET /v1/metrics/revenue`, 12 rpm, demo key) and Redis
> push invalidation. Driver: `scripts/benchmark_throughput_realpath.py --count 720000 --pace-eps 50`.  
> Raw artifacts captured off-stand: driver report JSON, 258 one-minute sampler rows
> (RSS/FD/lag/counters), hourly log. Stand host clock runs ~10 h fast; durations are
> relative and unaffected.

## Headline

| Question the soak asks | Answer |
|---|---|
| Does apply keep pace for hours? | **Yes.** 4 h at the delivered produce rate (~47 eps avg): hourly lag 71 / 44 / 134 / 0 — bounded, never trending. |
| Bridge memory / FD leak? | **None.** RSS sawtooths 38→114→11 MB (GC), FDs pinned at 94–95 for 4 h. |
| Lag death spiral? | **None.** Peak lag 2 915 during the one fault window, drained back to 0; the >5 000 alert never fired. |
| Loss or double-apply in the serving path? | **Zero.** ClickHouse distinct orders = 685 614 = 682 679 soak + 2 935 pre-existing, exactly the bridge's applied counter. |
| Cache drift after 4 h of writes + push invalidation? | **Zero.** API metric `1 540 429 855.37 ₽` == direct deduplicated ClickHouse sum `1 540 429 855.37 ₽`. |
| Batch amortization sustained? | **Yes.** 6 387 non-empty batches / 682 679 events → **mean ~107 events per batch**. |

## The fault it survived (this is the useful part)

At t≈194 min, under rising VM memory pressure, Redis publish calls had been
timing out intermittently (5 warnings over 2 h, invalidation is fire-and-forget
by design), then **one batch apply failed and the Kafka group session timed
out** (coordinator unresponsive ~45 s) — the consumer revoked and rejoined.
Recovery was fully automatic:

- offsets had not advanced past the failed batch → the 256-event batch was
  redelivered;
- the journal guard collapsed **all 256 as duplicates** (`duplicates_total`
  +256, exactly) — no loss, no double-apply;
- applied==consumed resumed immediately; `apply_failures_total` ended at **1**.

This live-exercises the delivery semantics documented in
[serving-bridge.md](../serving-bridge.md) under a real (not injected) fault.

## Produced vs applied — honest accounting

Driver queued 720 000 events; the bridge applied **682 679 (94.8 %)**. The
shortfall is on the **driver side, not the serving path**: rdkafka reported
**13 488 messages still in queue/transit at termination** (the driver's
`flush(30)` timed out), and the rest is consistent with producer-queue message
expiry during the t≈194 min broker-pressure window — the driver produces
fire-and-forget with no delivery callbacks, so undelivered events are not
individually attributable. Everything that reached `events.validated` was
applied exactly once (final lag 0, counters reconcile against the store).
Follow-up noted below.

## API under the same 4 h

- Reads: hour 1 = 720/720 OK; hour 2 = 1 424 OK + 16 client-side timeouts
  (during the pressure window); 0×429, 0×5xx observed throughout.
- **Finding — API process RSS grows without bound: 175 MB → 1.67 GB peak
  (~1.17 GB after load stopped)** over 4 h of 12 rpm reads with the ClickHouse
  backend and demo mode. The bridge stayed flat under far more work, so this
  is specific to the API process. Tracked as a GitHub issue; not a blocker for
  the write path, but it fails the "no leak" bar for the API and caps
  multi-day uptime on small hosts.

## Alert thresholds validated

| Signal | Threshold used | Verdict |
|---|---|---|
| `agentflow_bridge_consumer_lag` | alert > 5 000 | Right order of magnitude: real fault peaked at 2 915 and self-recovered; > 5 000 would indicate a genuine spiral. |
| `agentflow_bridge_apply_failures_total` | alert on any increase | Fired once, correctly; a single failure with offset hold + journal-guarded replay is recoverable, sustained growth is not. |
| Bridge RSS / FDs | trend over hours | Flat is the healthy signature (94–95 FDs). |
| API RSS | monotonic growth > ~200 MB/h | Fires on the real leak found here. |

## Environment ceiling (repeat of the q14 note)

The 6 GiB VM runs Kafka + ClickHouse + Flink (1 TM) + MinIO + Redis at
~90 % memory; hour-3+ pressure produced the Redis timeouts and the single
Kafka session timeout above. This is the stand's ceiling, not the product's —
but it is exactly the environment a small self-hosted deploy would see, so the
recovery behavior is the meaningful result.

## Follow-ups

1. **API RSS leak** — reproduce locally (unit-scale), profile allocators
   (usage writer queue, metric cache, CH client sessions are the suspects).
2. Driver hardening for future soaks: delivery callbacks + delivered-count in
   the report (attribute producer-side loss exactly), longer final flush.
3. Reader/sampler drift: hourly reader summaries slipped after client-side
   timeouts; harmless, but timestamp the lines rather than counting iterations.

## Reproduce

```bash
# stand as in throughput-realpath-q14-2026-07-10.md (1 TM), bridge on :9108,
# API on :8000 (AGENTFLOW_DEMO_MODE=true AGENTFLOW_SEED_ON_BOOT=true, CH backend)
.venv/bin/python scripts/benchmark_throughput_realpath.py \
  --bootstrap 127.0.0.1:19092 --count 720000 --pace-eps 50 \
  --bridge-metrics http://127.0.0.1:9108/metrics --latency-samples 0 \
  --catchup-timeout-seconds 900 \
  --report-json /tmp/soak-report.json --report-md /tmp/soak-report.md
# plus a 60 s sampler (RSS/FD/lag/counters) and a 5 s API reader — see the
# hourly/CSV artifacts referenced above.
```
