## TASK 1 — Competitive Analysis

**Цель:** documented positioning vs прямых конкурентов и заменителей. BCG §1.2 P3.

### `docs/competitive-analysis.md` (NEW)

**Структура:**

1. **Рынок и сегменты** (1 страница)
   - "Real-time data platforms for AI agents" — новая категория
   - Сегменты: streaming DW, real-time OLAP, cache layers, serverless data APIs

2. **Прямые конкуренты** (таблица):

| Продукт | Позиционирование | Pricing | Сильные стороны | Слабые стороны | AgentFlow advantage |
|---------|------------------|---------|-----------------|----------------|---------------------|
| **Tinybird** | Real-time analytics API | per-GB ingested | Managed, SQL-first, mature | No agent-native DX, no contracts | Pydantic/TS type safety, contract versioning |
| **Materialize** | Streaming DB | per-core | Incremental view maintenance | Heavy infra, no direct API | Lighter, agent-focused |
| **RisingWave** | Streaming DB (OSS + cloud) | OSS + cloud | OSS, SQL-like | New, smaller ecosystem | Similar — differentiation через SDK+contracts |
| **MotherDuck** | Serverless DuckDB | Per-compute | DuckDB familiarity | Not real-time streaming | Streaming + serving, not query engine |
| **Rockset** | Real-time OLAP | Acquired by OpenAI | Agent-native focus after OAI | Closed platform post-acquisition | Open-source alternative |

3. **Заменители** (таблица):

| Продукт | Когда заменяет AgentFlow | Когда не заменяет |
|---------|--------------------------|-------------------|
| Supabase Realtime | Simple CRUD с realtime | Агрегации, contracts, SLA |
| Firebase | Mobile-first, реалтайм базы | Сложные queries, аналитика |
| Custom Kafka+ClickHouse | Внутренний e-commerce | Team без DE expertise |
| PostgreSQL + pg_trgm | Simple search | Streaming, multi-tenant SLA |

4. **Positioning statement** (1 абзац):
   > AgentFlow — единственная real-time data platform **agent-native by design**: Pydantic-contracts, dual-SDK с паритетом, built-in contract versioning, freshness SLA. Для команд 5-50 чел. строящих AI-агенты для e-commerce/SaaS. Против Tinybird — open-source + agent-DX; против Materialize — простота; против ручного Kafka+ClickHouse — DX + TTP (time-to-production).

5. **Anti-positioning**:
   - НЕ для: BI dashboards (→ Metabase, Looker)
   - НЕ для: OLAP на петабайтах (→ Snowflake, ClickHouse)
   - НЕ для: simple CRUD (→ Supabase, Firebase)

## TASK 2 — Security Audit Report

**Цель:** формальный документ по security posture для enterprise due diligence. Консолидация того, что уже реализовано.

### `docs/security-audit.md` (NEW)

**Структура:**

1. **Executive Summary** (0.5 страницы)
   - Threat model: untrusted tenants, AI agents как clients, public API
   - Compliance posture: no PII storage without masking, audit trail, rate limiting

2. **Authentication & Authorization**
   - API key с bcrypt hashing (cost 12+)
   - Rotation: grace period, auto-revoke, admin endpoint
   - Tenant isolation: schema-per-tenant в DuckDB
   - Admin endpoints: отдельный `X-Admin-Key`
   - Evidence: `src/serving/api/auth/`, тесты в `tests/integration/test_rotation.py`

3. **Input Validation**
   - Pydantic schemas на всех endpoints
   - Cross-field validators (order.total == sum(line_items))
   - Contract versioning с deprecation policy
   - Evidence: `src/ingestion/schemas/events.py`, `tests/unit/test_event_schemas.py`

4. **SQL Injection Protection**
   - Parameterized queries на всех hot paths (v8)
   - `sqlglot` AST validator для NL→SQL (allowlist tables, no DDL/DML)
   - `_scope_sql` через sqlglot — работает с CTE/subqueries
   - `# nosec B608` только в documented trusted paths (catalog-provided identifiers)
   - Evidence: `src/serving/semantic_layer/query/`, `sql_guard.py`, 32 injection tests

5. **Rate Limiting & Abuse Protection**
   - Per-key + per-IP rate limits (Redis-backed, fallback in-memory)
   - Circuit breaker в SDK (v14)
   - Retry-After headers respected

6. **Data Protection**
   - PII masking middleware (configurable per tenant)
   - Security headers middleware (HSTS, CSP, X-Frame-Options)
   - TLS termination assumed at edge (ALB/nginx)

7. **Supply Chain**
   - Dependency scanning (Safety в CI)
   - Image scanning (Trivy в CI)
   - Bandit baseline — только `B310` historical + documented `# nosec`
   - Evidence: `.github/workflows/security.yml`, `.bandit-baseline.json`

8. **Operational Security**
   - Secrets в AWS Secrets Manager (Terraform)
   - Key rotation automation (v8)
   - Audit log: `usage.duckdb` tracks all API calls per-key
   - Incident runbook: `docs/runbook.md`

9. **Known Limitations**
   - Нет automated secret rotation (BCG I4) — manual rotation через admin endpoint
   - Нет external pen-test (рекомендация: запустить перед GA)
   - Chaos smoke на PR, full chaos — scheduled (не real-time detection)

10. **Compliance Readiness**
    - GDPR: PII masking, right-to-erasure через tenant DELETE endpoint (TODO — отсутствует)
    - SOC 2: audit trail + access controls готовы, нужен внешний аудитор
    - HIPAA: **не готов** — не encrypted at rest в DuckDB (только в managed backend)

### Формат блока (per endpoint)

```markdown
## GET /v1/entity/{entity_type}/{entity_id}

**Use case:** AI support agent needs latest order state to answer customer.

**curl:**
\`\`\`bash
curl -H "X-API-Key: $KEY" \\
  https://api.agentflow.dev/v1/entity/order/ORD-20260401-1001
\`\`\`

**Python:**
\`\`\`python
from agentflow import AgentFlowClient
client = AgentFlowClient("https://api.agentflow.dev", api_key=KEY)
order = client.get_entity("order", "ORD-20260401-1001")
print(f"Status: {order.data['status']}, freshness: {order.freshness_seconds}s")
\`\`\`

**TypeScript:**
\`\`\`typescript
import { AgentFlowClient } from 'agentflow';
const client = new AgentFlowClient("https://api.agentflow.dev", KEY);
const order = await client.getEntity("order", "ORD-20260401-1001");
console.log(`Status: ${order.data.status}, freshness: ${order.freshnessSeconds}s`);
\`\`\`

**Response 200:**
\`\`\`json
{"entity_type": "order", "entity_id": "...", "data": {...}, "freshness_seconds": 2.3}
\`\`\`

**Errors:**
- `404`: entity не найден
- `401`: отсутствует или неверный API key
- `403`: tenant изоляция
- `429`: rate limit, см. `Retry-After`
- `503`: table не материализована yet

**Rate limit:** 100 rpm per key (default), настраивается per-tenant.

## TASK 5 — Hosted demo config

**Цель:** публичный `demo.agentflow.dev` где потенциальный пользователь может погонять API без локальной установки.

**Платформа:** Fly.io (простая, есть free tier, быстрый deploy) ИЛИ Railway.

### `deploy/fly/fly.toml` (NEW)

```toml
app = "agentflow-demo"
primary_region = "fra"

[build]
  dockerfile = "../../Dockerfile.api"

[env]
  DUCKDB_PATH = "/data/agentflow-demo.duckdb"
  AGENTFLOW_DEMO_MODE = "true"
  AGENTFLOW_SEED_ON_BOOT = "true"

[[services]]
  internal_port = 8000
  protocol = "tcp"
  auto_stop_machines = true
  auto_start_machines = true
  min_machines_running = 0

  [[services.ports]]
    port = 80
    handlers = ["http"]
  [[services.ports]]
    port = 443
    handlers = ["tls", "http"]

  [services.http_checks]
    interval = "30s"
    path = "/v1/health"

[[mounts]]
  source = "agentflow_demo_data"
  destination = "/data"
```

### Demo mode в `src/serving/api/main.py`

Если `AGENTFLOW_DEMO_MODE=true`:
- Общий public API key `demo-key` с жёстким rate limit (60 rpm)
- Seed данные автоматически при старте
- Disable admin endpoints
- Read-only (отключить POST /v1/batch)

### `deploy/fly/README.md` — инструкция развёртывания

```markdown
# Fly.io Demo Deployment

## Initial setup (one-time)
1. `fly auth login`
2. `cd deploy/fly && fly launch --copy-config --no-deploy`
3. `fly volumes create agentflow_demo_data --size 1 --region fra`
4. `fly secrets set DEMO_API_KEY=<random-key>`

## Deploy
```bash
fly deploy -c deploy/fly/fly.toml --dockerfile ../../Dockerfile.api

```bash

```
```
