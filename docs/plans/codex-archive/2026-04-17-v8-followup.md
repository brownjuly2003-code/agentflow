## TASK 1 — Fix авто-revoke после grace period

**Симптом:**
```
FAILED tests/integration/test_rotation.py::test_expired_grace_period_revokes_old_key_automatically
assert 200 == 401
```

**Что происходит в тесте** (`tests/integration/test_rotation.py:191-226`):
1. Фикстура `expiring_client` (line 49) выставляет `AGENTFLOW_ROTATION_GRACE_PERIOD_SECONDS=1` и создаёт **новый** `AuthManager`, который присваивается в `c.app.state.auth_manager`.
2. POST `/v1/admin/keys/{key_id}/rotate` — успешно ротирует.
3. Тест в течение 3 секунд опрашивает GET `/v1/metrics/revenue` со СТАРЫМ ключом, ожидая 401.
4. Старый ключ продолжает работать (200) — auto-revoke timer не срабатывает.

**Цепочка автовозврата** (как должна работать):
- `KeyRotator.rotate_key` (`key_rotation.py:120-148`) вызывает `self._manager.load()` в конце.
- `AuthManager.load` (`manager.py:140-169`) на line 164-165: для каждого key с активным grace period → `schedule_rotation_cleanup(item)`.
- `KeyRotator.schedule_rotation_cleanup` (`key_rotation.py:304-314`) запускает `threading.Timer(delay, self.expire_previous_key, ...)`.
- `expire_previous_key` → `revoke_old_key` → clear `previous_key_hash` → `_manager.load()`.

**Вероятные причины регрессии (проверить ВСЕ):**

### Гипотеза A: фикстура создаёт второй AuthManager, таймер живёт в первом
В `expiring_client` fixture:
```python
with TestClient(app) as c:                       # ← TestClient запускает lifespan → app создаёт AuthManager #1
    manager = AuthManager(...)                   # ← Создаётся AuthManager #2
    manager.load()                               # ← Таймер планируется в #2, НЕ в том, что использует роутер?
    c.app.state.auth_manager = manager           # ← Подменяем
    yield c
    manager.shutdown()
```
Если admin router в `routers/admin.py` берёт `auth_manager` через DI, не из `app.state` — будет использоваться #1 (без подменённого grace=1s), и rotate отработает на нём, но проверки в `manager.authenticate` идут через #2.

**Проверка:** `grep -n "auth_manager" src/serving/api/routers/admin.py src/serving/api/auth/middleware.py` — увидеть, как разные пути получают manager.

### Гипотеза B: `_runtime_plaintext_by_hash` несёт старый plaintext
После initial load старый ключ был в `keys_by_value` (plaintext в YAML). После rotate YAML перезаписан на hash-only, `_runtime_plaintext_by_hash[new_hash] = new_key`. Но если где-то старый plaintext попадает в `_runtime_plaintext_by_hash` или `keys_by_value` — он остаётся даже после revoke + load.

**Проверка:** добавить временный print в `manager.load()` после line 168: `print("after load keys_by_value:", list(self.keys_by_value.keys()))`. Прогнать тест — посмотреть что внутри после revoke.

### Гипотеза C: timer не запускается из-за shutdown / lifecycle
В split-версии `AuthManager.__init__` или фикстуре `manager.shutdown()` вызывается до того как Timer успевает выполниться.

**Проверка:** `grep -n "shutdown\|cancel_rotation" src/serving/api/auth/manager.py src/serving/api/auth/key_rotation.py`.

### Гипотеза D: `expire_previous_key` ловит исключение тихо
В `key_rotation.py:316-328` — `except Exception` без re-raise. Если `revoke_old_key` падает (например `KeyError` из-за того что `_keys_by_id` пуст после shutdown) — лог пишется, но тест не видит ошибку.

**Проверка:** временно убрать try/except в `expire_previous_key`, прогнать тест — увидеть traceback.

### Шаги исправления

1. Воспроизвести: `pytest tests/integration/test_rotation.py::test_expired_grace_period_revokes_old_key_automatically -xvs`
2. Применить диагностические prints из гипотез A→B→D, найти реальную причину
3. Исправить (наиболее вероятно — гипотеза A: admin router использует не тот manager)
4. Удалить prints
5. Запустить весь rotation suite: `pytest tests/integration/test_rotation.py -v` — все 6+ тестов должны пройти
6. Запустить полный suite: `pytest tests/unit tests/integration --tb=line -q` — должно быть 427+ passed, 0 failed (с поднятым Redis — см. Task 2)

### Типичные узкие места и фиксы

1. **bcrypt verify на каждом auth check** — вызывается при каждом запросе. Фикс: in-memory cache `(api_key_hash, plaintext) → tenant_key` с TTL 60s в `auth/manager.py`.
2. **DuckDB connection pool слишком мал** — `db_pool.py` создаёт N connections. Если N=4 а concurrent=50 — очередь. Фикс: автоподбор `min(2*cpu_count, 16)`.
3. **JSON serialization в Pydantic** — для больших entity payloads. Фикс: `orjson` через FastAPI ORJSONResponse.
4. **Middleware overhead** — 6 слоёв middleware. Профилировать через `--include-time-spent`. Если auth/security middleware основная доля — кешировать verification result в request scope.

### Изменения

- НЕ трогать архитектуру. Только tuning constants и точечный кеш.
- Каждое изменение — отдельный коммит с микро-benchmark в commit message.
- Цель: p50 < 100ms для `/v1/entity/*` (другие endpoints не блокируют релиз).
