# Ответ на промпт раунда «событие → живая метрика»

Дата: 2026-06-06. Раунд выполнен **целиком, все четыре направления A-D**.
Всё лежит локально на ветке `feat/event-metric-freshness` (5 коммитов поверх main,
main не тронут). В origin НЕ запушено — push и PR ждут твоего «давай».

---

## Короткий итог

Ось «событие → живая метрика» теперь — измеренное свойство системы, а не нарратив:

> **Событие, вошедшее в пайплайн, отражается в `GET /v1/metrics/*` через
> 1.08 с (медиана) / 1.97 с (p95)** на боевых настройках по умолчанию.
> Настройкой поллера это ужимается до **210 мс p50**. Обычный TTL-кэш
> (то, что даёт «BI поверх реплики» с кэшем) на том же пайплайне сидел бы
> на ~15 с медианы.

Это число, которого нет ни у BI-поверх-реплики, ни в старой подаче — и оно
получено реальным замером, не придумано.

## A. Freshness-бенчмарк — сделан (коммит `0cb37d3`)

**Что меряется.** Полный путь: событие `order.created` проходит боевой код
пайплайна (схемная валидация → бизнес-валидация → обогащение → запись в
DuckDB, функция `local_pipeline._process_event`) — и засекается, через сколько
миллисекунд `GET /v1/metrics/revenue` по настоящему HTTP начинает отдавать
значение с этим заказом.

**Результаты** (82 замера, 0 таймаутов, канонический прогон):

| Режим | Что это | p50 | p95 | max |
|---|---|---|---|---|
| event_driven | прод-дефолты: кэш TTL 30 с + событийная инвалидация (поллер 2 с) | **1.08 с** | **1.97 с** | 2.06 с |
| fast_poll | то же, поллер ужат до 0.25 с | 210 мс | 332 мс | 382 мс |
| ttl_only | инвалидация выключена, чистый TTL (замер на TTL 5 с) | 2.72 с → **~15-16 с** при прод-TTL 30 с (линейное масштабирование) | | |
| no_cache | кэш выключен, каждый запрос в базу | 52 мс (пол; ограничен шагом опроса 25 мс) | | |

**Почему верить числам:** скрипт `scripts/benchmark_freshness.py`
воспроизводим одной командой (`python scripts/benchmark_freshness.py`),
методика зафиксирована в `docs/freshness-benchmark.md`. Ключевые честности:
кэш — боевой класс `QueryCache` (Redis-семантика через fakeredis, новая
dev-зависимость в extra `[load]`); чтения — авторизованным tenant'ом (без
ключа кэш вообще не работает — см. находку №3); API поднят в том же процессе,
потому что DuckDB пускает писателя только из одного процесса — так и устроен
прод-лейаут.

## B. Lineage «какие события двигают метрику» — сделан (коммит `4a40e8b`)

У каждой из 6 метрик в каталоге теперь явные поля `source_events` и
`source_table`, выверенные по реальному коду записи:

- `revenue`, `order_count`, `avg_order_value` ← `order.created/updated/cancelled` → таблица `orders_v2`
- `conversion_rate`, `active_sessions` ← `click`, `page_view`, `add_to_cart` → `sessions_aggregated`
- `error_rate` ← все 10 типов событий → `pipeline_events`

Отдаётся через `/v1/catalog` (добавлением полей — ничего не сломано) и через
типизированный SDK (`CatalogMetric`). Тест `tests/unit/test_catalog_lineage.py`
пинит карту против enum'а событий и обоих хендлеров каталога — расползтись
молча она не может.

## C. Контракты на все 6 метрик — сделаны (коммит `68fb604`)

Было: контракт только у `metric.revenue`. Стало: v1-контракт у каждой метрики,
сгенерирован штатным `scripts/generate_contracts.py` (CI-проверка `--check`
зелёная). В контрактах теперь:

- поля ответа (value/unit/window/computed_at) с unit'ами;
- `source_events`/`source_table` из направления B;
- блок `freshness`: **бюджет staleness 2.5 с p95** — это обещание (budget),
  опирающееся на замеренные 1.97 с p95, со ссылкой на бенчмарк; не пересказ
  числа.

Старые контракты (`order.v1/v2`) — байт-в-байт без изменений; registry читает
новые ключи аддитивно. Тест `tests/unit/test_metric_contracts.py` держит
контракт↔каталог в синхроне (lineage, unit'ы, версии).

## Находки по дороге (записаны, НЕ чинились втихую — нужны отдельные решения)

1. **Дыра в прод-оси: событийная инвалидация кэша метрик работает только если
   у tenant'а зарегистрирован хотя бы один активный webhook.** Поллер,
   который замечает новые события, обходит только тенантов с вебхуками. Без
   единого вебхука кэш не инвалидируется вообще → метрики живут на чистом
   TTL 30 с. Бенчмарк для честного замера регистрирует «сторожевой» вебхук с
   фильтром, который ничего не матчит. Кандидат-фикс: отвязать скан
   инвалидации от вебхуков (сканировать всегда, когда настроен кэш).
   Это прямо касается главного клейма — стоит решить до публичной подачи.

2. **`src/serving/api/main.py:305` на импорте мутирует общий роутер** —
   вырезает `GET /catalog` из `agent_query`-роутера, чтобы выигрывал более
   богатый `/v1/catalog` из main.py. Из-за этого поведение зависит от порядка
   импортов (я поймала 404 в тестах только под полным прогоном), а
   `agent_query.get_catalog` в проде — мёртвый код.

3. **Гоча для любых будущих замеров: кэш метрик — tenant-scoped.** Если есть
   `config/tenants.yaml`, анонимный запрос (без API-ключа) кэш полностью
   обходит. Мерять производительность/freshness — только авторизованным.
