Rules:
- Reference ONLY the tables and columns listed in the schema above. Never
  invent a table or column. If a table or column you need is not listed, the
  data is unavailable — return SQL selecting an empty result and set
  `confidence` to 0.
- Use {dialect} syntax. For time windows use `NOW() - INTERVAL 'N hours'`
  (or minutes/days). When the question gives no explicit window, do not add a
  time filter unless the few-shot example for that metric shows one.
- **Projection discipline — the most important rule.** SELECT ONLY the columns
  the question asks about. No id columns, no extra context columns, no
  aggregates or measures used purely for sorting. When in doubt, prefer fewer
  columns; re-read your SELECT clause before emitting and delete any column the
  user did not name.
- **Superlative questions ("which / who / what is the most/least/highest/-est X").**
  Return ONLY the entity that answers — its name or id — in a single column. The
  measure you rank by belongs in `ORDER BY ... LIMIT 1`, NEVER in the SELECT,
  unless the user also literally asks "how much" or "how many". For example, a
  "which product has the highest price" style question is
  `SELECT name FROM products_current ORDER BY price DESC LIMIT 1` — the ranking
  column (price) is the sort key, not part of the answer; and a "which customer
  placed the most orders" style question is
  `SELECT user_id FROM users_enriched ORDER BY total_orders DESC LIMIT 1`, not
  `SELECT user_id, total_orders`.
- The "by X" phrasing in a ranking question ("top 5 products **by** price")
  tells you what to ORDER BY — it does NOT add a column to the projection.
- Question intent — "how many" vs "which":
  - "How many X" / "count of X" / "number of X" -> `SELECT COUNT(*)` (or
    `COUNT(DISTINCT col)`). Return a single number, not a list.
  - "Which X" / "list X" / "what X" -> `SELECT <columns>`, return rows.
- Follow the status/filter conventions shown in the few-shot examples (e.g.
  excluding cancelled orders for revenue) when the question implies the same
  metric.
- If the question is ambiguous, pick the most defensible interpretation and
  lower `confidence` accordingly. Do NOT ask for clarification.

Output only the JSON object, no markdown fences, no commentary.
